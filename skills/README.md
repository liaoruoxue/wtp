# wtp Agent Skills

This directory contains agent-facing skills for using `wtp` as a workspace orchestration tool.

These skills are not for developing `wtp` itself. They are for helping coding agents such as Codex, Claude Code, and Cursor use `wtp` consistently and safely while working across multiple repositories.

## Included skills

- `wtp-workspace-operator` — create, inspect, reuse, and remove workspaces
- `wtp-repo-attach` — attach the current repo or another repo into a workspace
- `wtp-safe-cleanup` — safely eject worktrees and remove workspaces

## Design goals

- Prefer predictable command sequences over improvisation
- Distinguish clearly between `wtp switch` and `wtp import`
- Avoid relying on `wtp cd` for agent execution flows
- Treat dirty worktrees and `--force` as explicit safety boundaries
- Return workspace paths and repo layout in a way agents can continue using directly

## Recommended usage model

An agent should normally:

1. Discover current state with `wtp ls`, `wtp status`, and repository context
2. Reuse an existing workspace when appropriate
3. Use `wtp switch` for the current repository
4. Use `wtp import` only when already operating inside a workspace and attaching another repository
5. Use `wtp status` before cleanup actions
6. Avoid `--force` unless the user explicitly asked for it

## Why `wtp cd` is not the primary agent path

`wtp cd` depends on shell integration via `WTP_DIRECTIVE_FILE` and a wrapper generated by `wtp shell-init`. Humans can use that comfortably in an interactive shell, but agents should usually prefer plain directory paths and set their own working directory directly.
